# Manus MVP 需求文档

## 1. 文档目标

本文档用于定义第一阶段 Manus MVP 的产品范围、核心概念、功能需求和非功能需求，作为产品、设计和研发的统一基线。

本阶段目标不是复刻完整 Manus，也不是实现 Codex 式多 agent 协作平台，而是验证一个单 agent 的通用任务执行闭环是否成立。

## 2. 产品目标

### 2.1 核心目标

用户可以在一个对话中发起任务，系统能够：

1. 为任务创建独立执行环境。
2. 使用浏览器、代码、文件等工具执行多步任务。
3. 将执行过程和产物回传给前端。
4. 将结果沉淀到项目目录中。

### 2.2 成功标准

满足以下条件即视为 MVP 成立：

1. 用户可在前端创建项目并看到项目目录。
2. 用户可在对话中发起一个复杂任务。
3. 系统可自动执行多步流程并展示状态。
4. 系统可生成至少一种可交付产物，如 Markdown 报告、网页代码或导出文件。
5. 任务结束后产物可保存到项目目录中。

## 3. 非目标

本阶段不做以下内容：

1. 多 agent 并行协作。
2. 团队协作、权限体系、组织管理。
3. 插件市场或完整 MCP 生态。
4. 移动端、桌面端客户端。
5. 复杂长期记忆网络。
6. 复杂部署平台、网站正式托管能力。
7. 实时多人协同编辑。

## 4. 核心概念模型

### 4.1 Conversation

对话是用户与系统交互的界面容器，用于承载聊天消息和任务入口。

- 一个对话可关联一个项目。
- 一个对话可发起多个任务。
- 对话本身不绑定执行环境。

### 4.2 Project

项目是长期工作空间，用于承载目录结构、正式文件和任务产物。

- 一个项目可包含多个目录和文件。
- 一个项目可关联多个任务。
- 一个项目可被一个或多个对话引用。

### 4.3 Task

任务是系统执行的基本业务单元，代表一个明确目标，例如：

- 调研并生成报告
- 生成网页
- 分析文件并输出结论

任务应具备独立状态、日志、产物和重试能力。

### 4.4 Run

Run 是任务的一次执行实例。

- 一个任务可有多次 Run。
- 用户点击“重试”时，应创建新的 Run。

### 4.5 Sandbox

Sandbox 是任务 Run 对应的独立执行环境。

- 一个 Run 对应一个 Sandbox。
- 同一任务内多个步骤共享同一个 Sandbox。
- 不采用“一个对话一个 Sandbox”的模型。

### 4.6 Artifact

Artifact 是任务执行过程中生成的中间或最终产物，如：

- 报告
- 截图
- 网页代码
- 数据文件

Artifact 可被用户或系统保存进项目目录，形成正式项目文件。

## 5. 用户场景

### 场景 A：研究报告

1. 用户创建项目。
2. 用户在对话中输入“调研某公司并生成报告”。
3. 系统创建任务和执行环境。
4. Agent 搜索网页、提取信息、整理内容。
5. 系统生成 Markdown 或 PDF 报告。
6. 报告出现在对话中，并可保存到项目目录。

### 场景 B：网页生成

1. 用户在项目中发起“生成一个 landing page”任务。
2. 系统在 Sandbox 中生成代码和预览资源。
3. 结果文件回写到项目目录。
4. 用户可继续发起新任务，如“加一个 pricing 页面”。

### 场景 C：基于项目资料继续执行

1. 用户上传 PDF、文档或已有代码到项目。
2. 新任务可通过 RAG 检索这些资料。
3. Agent 基于项目知识继续生成结果。

## 6. 功能需求

### 6.1 前端

#### 6.1.1 项目创建与展示

系统应支持：

1. 创建项目。
2. 展示项目目录树。
3. 查看文本、图片、PDF 等基本文件。

#### 6.1.2 对话界面

系统应支持：

1. 在对话中输入任务。
2. 展示任务执行状态。
3. 展示步骤日志、工具调用、关键截图。
4. 展示最终结果。

#### 6.1.3 任务面板

系统应支持：

1. 查看任务状态：待执行、执行中、成功、失败、取消。
2. 查看任务历史 Run。
3. 重新运行任务。

### 6.2 后端 API

系统应提供以下核心 API：

1. 项目管理 API。
2. 对话管理 API。
3. 任务创建与查询 API。
4. Artifact 查询与下载 API。
5. 项目文件查询 API。
6. 任务事件流 API，建议使用 WebSocket 或 SSE。

### 6.3 Agent Orchestrator

系统应支持一个基础的单 agent 执行循环：

1. 接收任务目标。
2. 生成计划。
3. 调用工具执行。
4. 观察工具结果。
5. 在必要时继续迭代，直到结束。
6. 输出最终结果和产物。

系统应支持：

1. 步骤级日志记录。
2. 失败重试。
3. 超时终止。
4. 最终结构化结果输出。

### 6.4 Sandbox 管理

系统应支持：

1. 为每个 Task Run 创建独立 Sandbox。
2. 为 Sandbox 挂载独立工作目录。
3. 执行 Python、Shell、Browser 等操作。
4. 任务结束后回收 Sandbox。

本阶段建议使用 Docker 作为 Sandbox 方案。

### 6.5 工具系统

本阶段必须具备工具系统，至少包含以下内建工具：

1. `browser.open`
2. `browser.click`
3. `browser.type`
4. `browser.extract`
5. `browser.screenshot`
6. `file.list`
7. `file.read`
8. `file.write`
9. `python.run`
10. `web.fetch`
11. `final.answer`

要求：

1. 所有工具拥有统一参数结构和返回结构。
2. 工具调用有日志。
3. 工具执行结果可被前端展示。

### 6.6 Skill

本阶段应支持轻量 Skill 机制，但不做完整技能平台。

Skill 可定义为一类任务模板，包含：

1. System Prompt
2. 推荐步骤
3. 允许工具集合
4. 输出格式要求

本阶段至少支持：

1. 调研报告类 Skill
2. 网页生成类 Skill
3. 文件分析类 Skill

### 6.7 RAG

本阶段应支持轻量 RAG，仅覆盖以下范围：

1. 项目文件 RAG
2. 用户上传资料 RAG

要求：

1. 支持 PDF、Markdown、TXT、网页内容等基础解析。
2. 支持按项目范围检索。
3. 检索结果仅作为上下文增强，不替代项目文件原文。

### 6.8 记忆系统

本阶段应支持轻量记忆，不做复杂长期记忆网络。

建议支持以下三类：

1. 对话摘要记忆
2. 项目级事实记忆
3. 任务级 working memory

要求：

1. 记忆与 RAG 分离。
2. 记忆应可结构化存储。
3. 项目记忆应可被新任务复用。

### 6.9 Artifact 与项目文件流转

系统应支持：

1. Sandbox 中间结果导出为 Artifact。
2. Artifact 保存到项目目录。
3. 项目目录显示来自任务的文件产物。

必须区分：

1. Sandbox 临时文件
2. Artifact 中间或结果文件
3. Project 正式文件

### 6.10 模型接入与路由

本阶段需要考虑多模型接入，但不要求一次性接入大量厂商。

本阶段目标是建立统一模型抽象，避免系统直接绑定某一个厂商 SDK。

#### 6.10.1 模型接入原则

1. 模型能力通过统一接口暴露。
2. 业务编排层不得直接依赖具体模型厂商 SDK。
3. 模型配置应支持后续新增云端、自托管和本地模型。

#### 6.10.2 第一阶段支持范围

本阶段建议支持以下接入类型：

1. `OpenAI-compatible`
2. `Anthropic-native`
3. `Embedding provider`

本地模型或自托管模型在第一阶段可优先通过 `OpenAI-compatible` 方式接入。

#### 6.10.3 第一阶段模型能力

统一模型层至少应支持以下能力：

1. 文本生成
2. 工具调用
3. 结构化输出
4. Embedding

视觉能力可作为可选能力预留，不作为第一阶段必需项。

#### 6.10.4 路由要求

本阶段只做轻量路由，支持：

1. 系统默认主模型
2. 系统默认 embedding 模型
3. 每个任务可指定模型
4. 记录模型调用日志和错误信息

本阶段不要求实现复杂的自动 fallback、成本优化调度和动态多模型策略。

## 7. 数据模型要求

本阶段至少应具备以下核心实体：

1. `project`
2. `project_node`
3. `conversation`
4. `message`
5. `task`
6. `task_run`
7. `sandbox_session`
8. `artifact`
9. `project_memory`
10. `conversation_summary`
11. `document_chunk`
12. `model_provider`
13. `model_profile`

推荐关系：

- `conversation` -> 多个 `task`
- `project` -> 多个 `task`
- `task` -> 多个 `task_run`
- `task_run` -> 一个 `sandbox_session`
- `task_run` -> 多个 `artifact`

## 8. 非功能需求

### 8.1 可观测性

系统应记录：

1. 任务状态流转
2. 工具调用日志
3. Sandbox 生命周期
4. 错误堆栈
5. 关键截图或执行快照

### 8.2 安全隔离

本阶段至少做到：

1. 每个 Task Run 独立 Sandbox。
2. Sandbox 工作目录隔离。
3. 受控网络访问。
4. 任务结束后销毁运行环境。

### 8.3 可恢复性

系统应支持：

1. 任务失败后查看失败原因。
2. 重试任务。
3. 保留任务运行日志和产物。

### 8.4 性能

MVP 阶段不追求极致性能，但需满足：

1. 创建任务后可在可接受时间内启动 Sandbox。
2. 前端可实时看到任务状态变化。
3. 单任务长时间运行时系统不丢状态。

## 9. 技术方案建议

### 9.1 推荐技术栈

- 前端：Next.js
- 后端 API：FastAPI
- 任务编排：Python Worker
- 队列：Redis
- 数据库：Postgres
- 向量检索：pgvector
- 对象存储：S3 或 MinIO
- Sandbox：Docker
- 浏览器自动化：Playwright
- 模型接入：OpenAI-compatible + Anthropic 抽象层

### 9.2 架构原则

1. 使用模块化单体，而不是一开始拆微服务。
2. 控制层与执行层分离。
3. Tool、RAG、Memory、Sandbox 均应保留清晰边界。
4. 为后续引入 MCP 和多 agent 预留接口，但不在本阶段做重实现。

## 10. 版本边界

### 10.1 第一阶段必须完成

1. 项目创建与目录展示
2. 对话发起任务
3. 单 agent 多步执行
4. 每个 Task Run 独立 Sandbox
5. 基础工具系统
6. Artifact 导出与项目沉淀
7. 轻量 RAG
8. 轻量记忆

### 10.2 第二阶段再考虑

1. 多 thread
2. 多 agent
3. 团队协作
4. MCP 深度接入
5. 部署平台
6. 审批流
7. 更复杂的长期记忆系统

## 11. 最终判断

第一阶段 Manus MVP 的核心不是聊天，而是任务执行闭环。

本产品的最小正确模型为：

`Conversation -> Task -> Run -> Sandbox -> Artifact -> Project`

项目是结果沉淀容器，对话是交互容器，任务是执行容器，Sandbox 是运行容器。
