# PRD 架构设计指南

## 文档目的

本指南提供编写高质量 PRD（产品需求文档）的方法论，特别关注如何定义清晰的架构边界、设计一致的 API，以及制定可验证的成功标准。

本指南源自 Badgers MVP PRD 的编写经验，总结了在定义复杂系统时容易出现的边界模糊问题及其解决方法。

## 1. 资源边界划分原则

### 1.1 为什么需要明确边界

在定义多层资源模型时（如 Task/Run、Conversation/Task），如果边界不清晰，会导致：
- 前后端对资源属性的理解不一致
- API 设计自相矛盾
- 数据模型设计混乱
- 实施时反复纠结和返工

### 1.2 边界划分决策树

当定义两个相关资源（如 A 和 B）时，使用以下决策树判断某个属性/行为应该属于哪个资源：

**步骤 1：识别资源的核心职责**
- 资源 A 的核心职责是什么？（例如：Task = 定义目标）
- 资源 B 的核心职责是什么？（例如：Run = 执行实例）

**步骤 2：判断属性的生命周期**
- 这个属性会因为重试/重新执行而变化吗？
  - 是 → 属于执行层（Run）
  - 否 → 属于定义层（Task）

**步骤 3：判断属性的可变性**
- 这个属性在资源创建后会改变吗？
  - 是 → 属于状态层
  - 否 → 属于元数据层

**步骤 4：创建资源归属表**

为每个资源对创建一个明确的归属表：

| 属性/行为 | 归属于 | 判断依据 |
|----------|--------|---------|
| 目标描述 | Task | 定义层，不因重试而变 |
| 执行状态 | Run | 状态层，每次执行不同 |
| 创建时间 | Task | 元数据，创建时确定 |
| 开始时间 | Run | 执行层，每次执行不同 |

### 1.3 常见边界模式

**模式 1：定义 vs 执行**
- 定义层：目标、配置、模板
- 执行层：状态、日志、结果

**模式 2：容器 vs 内容**
- 容器层：组织结构、关联关系
- 内容层：具体数据、业务逻辑

**模式 3：临时 vs 持久**
- 临时层：执行中的中间状态
- 持久层：需要长期保留的结果

## 2. 交互流程设计原则

### 2.1 区分交互容器和执行单元

在设计用户交互流程时，必须明确区分：
- **交互容器**：用户与系统对话、讨论、澄清的地方
- **执行单元**：系统实际执行任务、产生结果的地方

**反模式：** 每条消息都触发执行
- 问题：用户无法自由讨论，每次发言都会启动任务
- 后果：用户体验差，资源浪费

**正确模式：** 明确的执行触发点
- 用户可以在交互容器中自由讨论
- 只有明确的触发动作（按钮、命令）才创建执行单元

### 2.2 交互流程设计检查清单

在设计交互流程时，回答以下问题：

1. **什么时候创建执行单元？**
   - [ ] 每条消息？
   - [ ] 用户明确点击按钮？
   - [ ] 用户输入特定命令？
   - [ ] 系统自动判断意图？

2. **用户如何知道执行已开始？**
   - [ ] 状态变化提示
   - [ ] 新的执行单元出现
   - [ ] 实时进度更新

3. **执行结果如何返回交互容器？**
   - [ ] 作为消息发布
   - [ ] 作为附件展示
   - [ ] 作为独立面板

4. **用户如何继续讨论结果？**
   - [ ] 在原交互容器中
   - [ ] 创建新的交互容器
   - [ ] 不支持讨论

### 2.3 API 设计对应

交互流程的边界必须体现在 API 设计中：

```
✅ 正确：分离交互和执行
POST /conversations/{id}/messages  # 发送消息，不执行
POST /tasks                        # 明确创建执行单元

❌ 错误：混合交互和执行
POST /conversations/{id}/messages
{
  "content": "...",
  "create_task": true  # 混淆了边界
}
```

## 3. 文件生命周期管理模板

### 3.1 为什么需要分层文件模型

在涉及文件操作的系统中，如果不明确区分不同层次的文件，会导致：
- 权限模型不清晰（agent 能否直接写项目文件？）
- 清理策略混乱（什么时候删除？）
- 存储成本失控（所有文件都永久保留？）

### 3.2 三层文件模型模板

定义文件系统时，使用以下三层模型：

**第一层：临时工作文件**
- **定义：** 执行过程中的中间文件
- **存储位置：** 执行环境内（如 Docker 容器）
- **生命周期：** 执行结束后 N 小时删除
- **访问权限：** 仅执行环境内可访问
- **用途：** 中间数据、临时脚本、调试文件

**第二层：可交付产物**
- **定义：** 值得保留的执行结果
- **存储位置：** 对象存储（S3/MinIO）
- **生命周期：** 保留 M 天（如 30 天），未保存则自动删除
- **访问权限：** 通过 API 下载
- **用途：** 报告、截图、生成的代码

**第三层：正式项目文件**
- **定义：** 用户明确保存的资产
- **存储位置：** 项目目录（数据库 + 文件系统）
- **生命周期：** 永久（直到用户删除）
- **访问权限：** 项目成员可访问，可被索引和检索
- **用途：** 项目文档、源代码、配置文件

### 3.3 文件流转规则

明确定义文件如何在三层之间流转：

```
第一层（临时）→ 第二层（产物）
- 触发条件：执行完成
- 流转方式：系统自动提取
- 选择标准：文件大小、类型、位置

第二层（产物）→ 第三层（项目）
- 触发条件：用户明确操作
- 流转方式：用户点击"保存到项目"
- 选择标准：用户判断价值

第三层（项目）→ 第一层（临时）
- 触发条件：新执行需要引用
- 流转方式：只读挂载或复制
- 选择标准：执行需要的上下文
```

### 3.4 权限模型定义

基于三层模型，明确定义各角色的权限：

| 角色 | 第一层（临时） | 第二层（产物） | 第三层（项目） |
|------|--------------|--------------|--------------|
| Agent | 读写 | 只能创建 | 只读（通过上下文） |
| 用户 | 不可访问 | 读、下载、保存 | 完全控制 |
| 系统 | 创建、清理 | 存储、清理 | 索引、检索 |

## 4. RESTful API 设计规范

### 4.1 资源层级设计原则

**原则 1：URL 应该反映资源的层级关系**

```
✅ 正确：清晰的层级
GET /projects/{project_id}/files
GET /tasks/{task_id}/runs
GET /runs/{run_id}/logs

❌ 错误：扁平化所有资源
GET /files?project_id=123
GET /runs?task_id=456
```

**原则 2：独立资源应该有独立端点**

如果资源可以独立存在和查询，应该有独立端点：

```
✅ 正确：Run 是一等资源
GET /runs/{run_id}
POST /runs/{run_id}/cancel

❌ 错误：Run 只能通过 Task 访问
GET /tasks/{task_id}/runs/{run_id}
POST /tasks/{task_id}/cancel  # 实际取消的是 run
```

### 4.2 Path Parameter vs Query Parameter

**Path Parameter 适用场景：**
- 资源标识符（ID）
- 层级关系中的父资源
- 不包含特殊字符的简单值

**Query Parameter 适用场景：**
- 过滤条件
- 排序、分页参数
- 可选参数
- 包含特殊字符的值（如文件路径）

**示例：**

```
✅ 正确：文件路径使用 query parameter
GET /projects/{project_id}/files/download?path=/reports/tesla.md
GET /projects/{project_id}/nodes/{node_id}/content

❌ 错误：文件路径作为 path parameter
GET /projects/{project_id}/files/{file_path}
# 问题：file_path 可能包含 /，导致 URL 解析错误
```

### 4.3 资源状态和操作的归属

**规则：状态和操作应该归属于拥有该状态的资源**

```
✅ 正确：状态属于 Run，操作也在 Run 上
GET /runs/{run_id}  # 返回 status, started_at
POST /runs/{run_id}/cancel

❌ 错误：状态属于 Run，但操作在 Task 上
GET /tasks/{task_id}  # 返回 status（实际是 run 的状态）
POST /tasks/{task_id}/cancel  # 实际取消的是 run
```

### 4.4 事件流和实时更新

**规则：事件流应该挂在产生事件的资源上**

```
✅ 正确：事件由 Run 产生
WS /runs/{run_id}/events

❌ 错误：事件由 Run 产生，但挂在 Task 上
WS /tasks/{task_id}/events
```

## 5. MVP 成功标准定义方法

### 5.1 功能清单 vs 假设验证

**反模式：功能清单式成功标准**

```
❌ 错误示例：
- ✅ 用户可以创建项目
- ✅ 用户可以创建任务
- ✅ 系统可以执行任务
- ✅ 用户可以查看结果
```

问题：
- 只关注功能是否实现，不关注是否解决问题
- 无法判断产品假设是否成立
- 容易陷入"功能越多越好"的陷阱

**正确模式：假设驱动的验证标准**

```
✅ 正确示例：
假设 1：用户理解任务委托模型
- 验证方法：观察 5 个用户的首次使用
- 成功标准：4/5 用户能在 5 分钟内创建任务
- 失败信号：用户不理解何时创建任务
```

### 5.2 定义核心假设的方法

**步骤 1：识别产品的核心价值主张**

问自己：这个产品的核心创新是什么？

例如：Badgers 的核心创新是"用户可以委托复杂任务给 AI，AI 在隔离环境中自主执行"

**步骤 2：将价值主张分解为可验证的假设**

| 价值主张 | 核心假设 |
|---------|---------|
| 用户可以委托任务 | 用户理解并采用任务委托模型 |
| AI 自主执行 | Agent 能可靠完成典型任务 |
| 隔离环境 | 用户能理解和信任执行过程 |
| 产生结果 | 产物有价值并被保存使用 |

**步骤 3：为每个假设定义验证方法**

每个假设需要回答：
1. **如何验证？** 观察、测量、询问
2. **成功标准是什么？** 具体的数字或行为
3. **失败信号是什么？** 什么情况说明假设不成立

### 5.3 假设验证模板

使用以下模板定义每个核心假设：

```markdown
#### 假设 N：[假设陈述]

**假设：** [详细描述假设内容]

**验证方法：**
- [具体的验证步骤]
- [需要收集的数据]
- [观察的用户行为]

**成功标准：**
- ✅ [可量化的指标 1]
- ✅ [可量化的指标 2]
- ✅ [可观察的行为]

**失败信号：**
- [什么情况说明假设不成立]
- [需要警惕的负面指标]
```

### 5.4 最小可行功能集

在定义了核心假设后，反推最小可行功能集：

**问题：** 要验证这些假设，最少需要哪些功能？

**原则：**
- 只包含验证假设必需的功能
- 明确标注 MVP 可接受的限制
- 区分"必须验证的"和"可以延后的"

**示例：**

```markdown
### 最小可行功能集

**必要功能：**（验证假设必需）
- ✅ 任务创建流程
- ✅ 实时执行更新
- ✅ 产物保存功能

**可接受的限制：**（不影响假设验证）
- 仅单代理（无多代理）
- 有限的技能模板（3 种）
- 基本错误处理（无自动恢复）
```

## 6. PRD 编写检查清单

在完成 PRD 初稿后，使用以下检查清单进行自查：

### 6.1 资源边界检查

- [ ] 每对相关资源都有明确的边界定义
- [ ] 创建了资源归属表，列出所有属性的归属
- [ ] 没有属性同时出现在两个资源中
- [ ] API 返回值与资源边界定义一致

### 6.2 交互流程检查

- [ ] 明确定义了执行单元的创建时机
- [ ] 交互容器和执行单元的 API 是分离的
- [ ] 用户可以在不触发执行的情况下讨论
- [ ] 执行结果如何返回交互容器有明确说明

### 6.3 文件生命周期检查

- [ ] 定义了文件的三层模型（或类似分层）
- [ ] 每层文件的存储位置、生命周期、访问权限都已明确
- [ ] 文件在各层之间的流转规则已定义
- [ ] 权限模型清晰（谁能读/写哪一层）

### 6.4 API 设计检查

- [ ] 资源层级在 URL 中清晰体现
- [ ] 独立资源有独立端点
- [ ] Path parameter 和 query parameter 使用正确
- [ ] 状态和操作归属于同一资源
- [ ] 事件流挂在产生事件的资源上

### 6.5 成功标准检查

- [ ] 定义了 3-5 个核心假设
- [ ] 每个假设都有验证方法
- [ ] 每个假设都有可量化的成功标准
- [ ] 每个假设都有失败信号
- [ ] 最小可行功能集是为验证假设服务的

## 7. 常见问题和解决方案

### Q1: 如何判断两个概念是否应该分离？

**判断标准：**
1. 它们的生命周期是否不同？
2. 它们的职责是否可以独立描述？
3. 用户是否需要分别操作它们？

如果以上任一问题答案为"是"，应该分离。

### Q2: API 设计和资源边界定义冲突怎么办？

**优先级：** 资源边界定义 > API 设计

如果 API 设计与资源边界定义冲突，应该修改 API 设计，而不是妥协资源边界。

### Q3: 成功标准太多怎么办？

**原则：** MVP 只验证 3-5 个最核心的假设

如果有超过 5 个假设，说明：
- 产品范围太大，需要缩小
- 某些假设不够核心，可以延后验证
- 某些假设可以合并

### Q4: 如何处理遗留系统的边界问题？

**策略：**
1. 在 PRD 中明确定义理想的边界
2. 在实施计划中标注与遗留系统的差异
3. 制定渐进式迁移计划
4. 不要在 PRD 中妥协边界定义

## 8. 使用本指南

### 8.1 PRD 编写流程

1. **定义核心概念** → 使用资源边界划分原则
2. **设计交互流程** → 使用交互流程设计原则
3. **设计文件系统** → 使用文件生命周期模板
4. **设计 API** → 使用 RESTful API 规范
5. **定义成功标准** → 使用假设验证方法
6. **自查** → 使用 PRD 编写检查清单

### 8.2 团队协作

- **产品经理：** 重点关注第 2、5 节（交互流程、成功标准）
- **架构师：** 重点关注第 1、3、4 节（资源边界、文件生命周期、API 设计）
- **开发人员：** 使用第 6 节检查清单验证 PRD 的可实施性

### 8.3 持续改进

本指南应该随着项目经验不断更新：
- 记录新的边界模糊问题及解决方案
- 补充新的设计模式和反模式
- 更新检查清单
