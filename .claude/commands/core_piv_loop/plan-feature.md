# 规划新任务

## 功能：$ARGUMENTS

## 任务目标

通过系统性的代码库分析、外部研究与策略规划，将一个功能需求转化为**完整的实施计划**。

**核心原则**：在这个阶段我们不编写代码。目标是产出一份上下文充分的实施计划，使 AI 代理可以一次性高质量完成实现。

**关键理念**：上下文就是一切。计划必须包含实现所需的全部信息，包括模式、必读文件、文档、验证命令等，确保执行代理第一次就能成功。

## 规划流程

### 阶段 1：功能理解

**深入分析功能：**

- 提炼要解决的核心问题
- 识别用户价值和业务影响
- 判断功能类型：新能力 / 增强 / 重构 / 缺陷修复
- 评估复杂度：低 / 中 / 高
- 梳理会受影响的系统与组件

**创建用户故事，或在用户已提供故事时进行完善：**

```text
作为 <用户类型>
我想要 <动作/目标>
以便 <收益/价值>
```

### 阶段 2：收集代码库情报

**使用专门代理并行分析：**

**1. 项目结构分析**

- 识别主要语言、框架与运行时版本
- 梳理目录结构与架构模式
- 确定服务 / 组件边界与集成点
- 定位配置文件（`pyproject.toml`、`package.json` 等）
- 找出环境初始化与构建流程

**2. 模式识别**（有利时使用专门子代理）

- 在代码库中搜索类似实现
- 识别编码约定：
  - 命名模式（CamelCase、snake_case、kebab-case）
  - 文件组织方式与模块结构
  - 错误处理方案
  - 日志模式与规范
- 提炼该功能领域的通用模式
- 记录需要避免的反模式
- 检查 `CLAUDE.md` 中的项目专属规则与约定

**3. 依赖分析**

- 列出与该功能相关的外部库
- 理解这些库是如何接入的（检查导入、配置）
- 在 `docs/`、`ai_docs/`、`.agents/reference` 或 `ai-wiki` 中查找相关文档（如果存在）
- 记录库版本与兼容性要求

**4. 测试模式**

- 识别测试框架与组织结构（`pytest`、`jest` 等）
- 查找类似测试示例作为参考
- 理解测试组织方式（单元测试 vs 集成测试）
- 记录覆盖率要求与测试标准

**5. 集成点**

- 找出需要更新的已有文件
- 确定需要创建的新文件及其位置
- 梳理路由 / API 注册模式
- 如适用，理解数据库 / 模型模式
- 如相关，识别认证 / 授权模式

**澄清不明确之处：**

- 如果此时需求仍不清晰，继续前先让用户澄清
- 获取明确的实现偏好（库、方案、模式）
- 在继续之前先解决架构层决策

### 阶段 3：外部研究与文档收集

**有利时使用专门子代理进行外部研究：**

**收集文档：**

- 研究最新库版本与最佳实践
- 查找带具体锚点的官方文档
- 定位实现示例与教程
- 识别常见陷阱与已知问题
- 检查破坏性变更与迁移指南

**技术趋势：**

- 研究当前技术栈的最佳实践
- 查找相关博客、指南或案例研究
- 识别性能优化模式
- 记录安全注意事项

**整理研究参考资料：**

```markdown
## 相关文档

- [库官方文档](https://example.com/docs#section)
  - 特定功能的实现指南
  - 原因：实现 X 功能所必需
- [框架指南](https://example.com/guide#integration)
  - 集成模式章节
  - 原因：说明组件如何连接
```

### 阶段 4：深度战略思考

**重点深入思考：**

- 这个功能如何融入现有架构？
- 关键依赖和执行顺序是什么？
- 可能出现哪些问题？（边界情况、竞争条件、错误）
- 如何对它进行全面测试？
- 是否存在性能影响？
- 是否有安全方面的考虑？
- 这个方案是否易于维护？

**设计决策：**

- 在备选方案中做出选择，并给出清晰理由
- 为可扩展性和未来修改而设计
- 如有需要，规划向后兼容
- 考虑可伸缩性影响
